## OWASP Top 10 취약점 이해 및 공격 방법(IDOR, SQLI, XSS, CSRF, LFI/RFI/SSRF...)

### IDOR (Insecure Direct Object References)

IDOR이란 주로 SS 컨트롤과 관련이 있다.

웹으로부터 오는 입력이 개체에 직접적으로 창조되어 접근할 때, 이러한 입력을 악용해 공격자가 본인의 고난을 초과하는 동작을 수행할 수 있는 경우를 말한다.

또, 사용자의 계정 설정 페이지에 접근하는 URL이 사용자 ID를 직접 참조하는 경우, 공격자는 URL의 사용자 ID 부분을 변경해 다른 사용자의 계정 설정 페이지에 접근할 수 있다.

(ex. 파라미터 값이 단순 숫자인 경우 간단하게 UID값을 조작해서 큰 영향을 주는 해킹가능)

- Horizontal Privilege Escalation - 동일한 권한 수준을 가진 다른 사용자의 데이터에 엑세스
- Vertical Privilege Escalation - 사용자 권한 → 관리자 권한

### XSS (Cross-site Scripting)

- Reflected XSS: 사용자에게 보내지는 링크에 악성 스크립트가 삽입된다. 공격자가 보낸 URL 링크를 클릭하면, 사용자의 브라우저에서 악성 스크립트가 실행된다.
- Stored XSS: 악성 스크립트가 서버에 저장된다. 그리고 매번 취약한 사이트에 접속할 때마다 악성 스크립트가 실행된다.
- DOM-based XSs: 페이지에 작성된 javascript에 의해서 발생하는 악성 스크립트이다. 때문에 사이트에 별도의 요청을 보내지 않아 탐지가 어려운 XSS이다.
- Other XSS: Blind XSS, Universal XSS, Server-side XSS, Mutated XSS…

#### **Reflected XSS** 시나리오

공격자가 보안이 취약한 웹사이트를 찾았다고 가정했을 때

이 공격자는 취약점을 이용해 사용자 정보를 빼돌릴 수 있는 악성 스크립트가 포함된 URL을 생성한다. (사용자가 이 URL에 접근하면 악성 스크립트가 실행됨)

이제 이 URL을 일반 사용자에게 전달 (보통 이메일이나 메시지 등을 통해 스팸 형태로 전달)

사용자는 이 메일을 통해 전달을 받은 URL 링크를 클릭하게 되고, 일반 사용자의 웹브라우저가 이 URL을 통해 취약한 웹사이트의 웹 서버로 요청을 보내게 된다. 서버는 이 요청을 처리하고 반환한다. 이때 응답 내용에는 악성 스크립트가 포함되어 있기 때문에 응답을 받은 일반 사용자의 브라우저에서는 이 악성 스크립트가 실행된다.

스크립트가 실행되면서 사용자의 민감한 정보를 공급자에게 전달하게 되고, 이렇게 공격자는 원하는 정보를 획득하게 된다.

#### Stored XSS 시나리오

공격자는 먼저 보안이 취약한 웹사이트를 찾아내고, 그곳의 게시판에 사용자 정보를 빼낼 수 있는 악성 스크립트를 삽입하는 게시글을 작성하게 된다. 

이후 일반 사용자가 공격자가 작성한 그 게시글을 읽으려 할 때, 웹서버는 해당 사용자에게 악성 스크립트가 포함된 게시글을 전달한다. 

그 다음 일반 사용자의 브라우저에서는 전달받은 게시글의 내용을 렌더링하는 과정에서 악성 스크립트가 실행된다. 

이렇게 실행된 악성 스크립트를 통해 사용자의 민감 정보가 공격자에게 전달된다. 

#### DOM-based XSS 시나리오

공격자는 우선 보안이 취약한 웹 페이지를 탐색하고, 해당 페이지에서 악성 스크립트가 실행될 수 있도록 조작된 URL 주소를 생성한다.

이 URL을 일반 사용자에게 전달한다. 이 URL 링크를 클릭하면…

웹 서버는 해당 URL에 대응하는 HTML 문서를 응답으로 보낸다. 사용자의 브라우저는 받은 HTML 문서를 해석하고, 필요한 스크립트를 실행하는 과정에서 공격자가 조작한 악성 스크립트도 같이 실행하게 된다. 

그 결과 악성 스크립트는 일반 사용자의 개인 정보를 탈취해 공격자의 서버로 전달한다. 

DOM-based XSS가 Reflected XSS와 다른 점은 모든 악성 행동이 사용자의 브라우저 상에서만 발생하므로 서버 로그를 확인해도 공격 흔적을 찾아내기 힘들다는 점이다. 

### CSRF/XSRF (Cross-site Request Forgery)

사용자의 인증 정보를 이용해 사용자의 의지와는 무관하게 공격자가 의도한 서비스 요청을 처리하는 공격 방법

#### 시나리오

1. 공격자가 사용자에게 악성 URL 이메일로 공유
2. 사용자 서버에서 은행 서버로 악성스크립트(ABC123 계좌에 1000원 전송해) 실행
3. 은행 서버에서 사용자 서버로 응답(이체 완료)
4. 공격자는 돈이 들어왔나?하며 계좌조회 가능

### SQL Injection

공격자가 원래 SQL 구문의 의미를 바꿔서 예상치 못한 동작이나 정보에 유출을 일으키는 공격방법이다. 

#### SQL Injection Techniques

1. Boolean-based blind SQL Injection
    
    조건에 들어가는 구문에 참 거짓에 대한 쿼리를 작성하는 방법
    
    코드에서 AND 1=1이 들어갔을 때는 항상 옳고, 유저 네임이 조회되지만 AND 1=0 구문이 들어가면 항상 틀리기 때문에 아무런 정보가 조회되지 않는 것을 의미한다.  
    
    `SELECT * FROM Users WHERE Username=’dom’ AND 1=1`
    
2. Error-based SQL Injection
    
    이 기법은 일부러 유해하지 않은 쿼리로 조작해 오류메시지가 반환되지 않게 하는 기법이다.
    
    `SELECT * FROM Users WHERE Username=’dom’ UNION SELECT 1,2,3--`
    
3. UNION query SQL Injection
    
    유니온을 사용해 두 쿼리를 합쳐 실행한다. 
    
    밑 코드에서 NULL이 들어간 이유는 첫 번째 나온 쿼리에서 조회되는 칼럼 회수를 맞추기 위해서이다.
    
    `SELECT * FROM Users WHERE Username=’dom’ UNION SELECT NULL, NULL, NULL, ...`
    
4. Stacked queries SQL Injection
    
    쿼리를 끊어내고 추가 쿼리를 삽입해서 실행한다.
    
    `SELECT * FROM Users WHERE Username=’dom’; DROP TABLE Users--`
    
5. Time-based blind SQL Injection
    
    Slip이나 Wait for 함수를 사용해서 조건에 따른 시간을 이용한 쿼리를 실행한다.
    
    `SELECT * FROM Users WHERE Username=’dom’ AND IF(1=1, SLEEP(10), FALSE)`
    
6. Inline queries SQL Injection
    
    Select Query 내에 동일한 Select Inline Query를 삽입해서 실행한다.
    
    `SELECT *, (SELECT User FROM Users WHERE UserID=1)FROM Users WHERE Username=’dom’`
    

### File Inclusion

Directory Traversal(파일을 읽는 기능을 악용해 시스템 파일을 탈취)와 달리 동적으로 파일을 읽거나 Include(소스코드 내 포함)하는 기능을 악용해 시스템 파일을 탈취하거나 임의 소스코드를 Include하도록 유도하는 유형이다. 

#### LFI (Local File Inclusion)

Local에 있는 파일을 읽는 공격이다. 이를 통해 공격자는 시스템 파일을 탈취하거나 미리 업로드 된 소스 코드를 인클로드 시켜서 이동한 코드를 실현하기도 할 수 있는 취약점이다.

#### RFI (Remote File Inclusion)

공격자가 웹 서버에서 자신이 원격으로 제어하는 파일을 포함시키는 방법을 이용한 공격이다. 이를 통해 공격자는 원격 서버에서 호스팅되는 파일을 웹 어플리케이션에 포함시키고 실행시킬 수 있게 된다. 

예를 들어 공격자는 자신의 원격 서버에 악성 코드를 포함한 파일(shell 코드)을 미리 준비해두었다고 가정했을 때 이후 취약한 웹 서버와 이 원격 파일을 포함하도록 만들면 공격자의 원격 파일이 웹 어플리케이션에 일부러 포함되면서 웹 서버에서 공격자의 코드가 실행되게 된다. 

### SSRF (Server-side Request Forgery)

공격자가 서버로 하여금 임의의 외부 요청을 전송하도록 만드는 방식이다.

#### 시나리오

1. GET /?url=http://127.0.0.1:21을 공격자가 웹 서버에 보냄
2. 웹 서버는 자기 자신에게 다시 요청
3. 웹 서버가 공격자에게 응답을 줌

## 웹 취약점의 영향 및 위협

### 시나리오 개발이 필요한 이유

예시

- XSS의 주요 목적
    - 세션 탈취
    - 하지만 http only 설정 등으로 세션 탈취가 안 된다면?
        - 아무런 시나리오를 세우지 않고 제보하는 XSS는 포상을 받지 못할 확률이 높음
        - test 페이지 등에서 개발자가 의도하지 않은 세션 노출이 있는지 확인
            - 세션 노출 페이지를 호출해서 세션 가져오기
            - 즉, 세션 탈취 시나리오가 가능해짐
        - 오픈 리다이렉트, 브라우저 자원 소진, 브라우저 제로데이 연계로 로컬 접근 등의 시나리오로 제보

XSS 취약점 & 세션정보 노출 취약점 → 세션정보 탈취 시나리오

### Self-XSS

XSS의 한 종류로 사용자가 본인의 웹 브라우저에서 악성 스크립트를 실행하도록 유도하는 공격 방법

이 방식은 공격자가 웹사이트에 직접 악성 스크립트를 삽입하는 것이 아니라 피해자가 스스로 자신에게 피해를 준 특징이 있다.

### 취약점 체이닝

2개 이상의 취약점을 연결하여 상대적으로 중요하지 않은 보안 이슈들을 결합해 강력한 공격 시나리오를 구축하는 방법을 말한다.

이런 체이닝 방식은 공격자에게 강력한 공격벡터를 제공한다. 공격은 일반적으로 하나의 취약점을 이용해 제한적인 권한을 얻지만 이러한 취약점들을 결합하면 그 효과가 훨씬 더 커진다.

#### XSS - Reflected

- Reflected XSS
    - 사용자의 입력 값이 응답 내용에 안전하지 않은 방식으로 포함될 때 발생하는 취약점
    - 사용자의 입력 값을 다시 되돌려 받는다는 느낌에 ‘반사된’이라는 표현을 사용함
- 예시

    - 결과) <h1><script>alert(1)</script><h1>
- 악용 시나리오

#### File Download - Path Traversal

- Path Traversal
    - 디렉터리 순회(Path Traversal) 취약점은 사용자가 조작할 수 있는 입력 값이 서버의 파일 시스템에 전달되고 안전하지 않은 방식으로 사용될 때 발생하는 취약점
    - 공격자는 조작 가능한 매개변수에 ../나 ..\와 같은 상위 경로에 접근하는 문자열을 입력해 웹 루트 디렉터리를 벗어나 접근이 제한된 상위 디렉터리에 접근해 민감 파일을 열람할 수 있게 됨
- 예시
    - `http://127.0.0.1/download?filename=../../../etc/passwd`
- 악용 시나리오
    - 소스코드 탈취, 서버 내 중요 민감 정보 탈취 등

#### File Downlaod - LFI

- LFI (Local File Inclusion)
    - LFI 취약점은 include할 페이지 경로가 적절히 필터링 되지 않았을 때 발생함
    - PHP에서 include 함수는 특정 경로 및 파일을 현재 페이지에 포함하고자 할 때 사용함
- 예시
    
    ```php
    <?php
    		$file = $_GET["file"];
    		if(isset($file)) {
    			include("pages/$file")
    		}else
    		{
    			include("index.php");
    		}>
    ```
    
- 악용 시나리오
    - 서버 파일 탈취, 소스코드 탈취, 민감 파일 탈취, 원격 코드 실행 등

#### Remote Code Execution

- 원격 코드/명령 실행
    - RCE 공격은 공격자가 대상 서버에서 임의 코드를 실행할 수 있는 공격이다.
- 악용 시나리오
    - 멀웨어/악성코드 배포, 서버 내 중요 데이터 탈취/파괴, DoS 공격 등
- 체이닝이란?
    - 체이닝은 여러 취약점을 연계하거나 단일 취약점과 다른 정상적인 기능들을 함께 사용해 취약점 파급도를 높이는 것을 의미함
    - 오픈소스 버그헌팅에서의 원격 코드 실행 취약점은 대부분 취약점 체이닝을 통해 발생함
