## 버그헌팅 소개

### 버그헌팅과 버그바운티

- Bug Hunting → SW/APP에서 버그를 찾는 활동
- Bug Bounty → 버그 헌팅의 보상 제도

### 버그헌팅을 해야하는 이유 (해커 관점)

1. 보안 강화 기여
    
    → 많은 사람들이 안전하게 서비스를 이용할 수 있도록 도움이 됨
    
2. 학습과 성장
    
    → 더욱더 전문적인 화이트해커 성장 가능
    
3. 가치 증명
    
    → 본인의 커리어를 높일 수 있는 발판이 될 수 있음
    

### 버그헌팅을 해야하는 이유 (기업)

Public 버그 바운티는 다수의 보안 전문가의 점검 대상이 되어, 짧은 시간 안에 많은 취약점을 발견할 수 있어서 효율적이다.

하지만 해커의 선악 파악이 불가하여 내부 헌팅과 외부 헌팅의 비율을 잘 조절해야한다.

### 버그바운티 포상금

포상금은 프로그램 별로 상이하며 취약점 위험도에 따른 테이블이 존재한다.

- Critical: Zero-Click RCE, Pre-Auth RCE
- High: Restricted RCE, SQL Injection
- Medium: Stored XSS, Logical Flow Bug
- Low: Reflected XSS, Open Redirect
- Informative: Information Disclosure

### 버그헌팅에 대한 오해 (해커 관점)

- 버그헌팅은 합법인가요?
    - 네. 사전에 버그헌팅 프로그램에서 허용된 범위를 정하고, 취약점을 찾아주기 때문에 합법이다. 다만, 허용된 범위 외의 버그헌팅을 불법이고, 사전에 협의되지 않은 범위에 대해 대비하지 못한 시스템에 문제가 생긴다면 처벌을 받을 수 있다.
- 코드 분석이나 시스템 구조, 모의해킹 기술을 잘 알아야 시작할 수 있지않나요?
    - 아니요. 모르더라도 버그헌팅을 시작할 수 있습니다. 하지만 기술을 잘 알수록 더 깊이있는 취약점을 찾을 수 있는 것 또한 사실이다.

### 버그바운티 프로그램에 대한 오해 (기업 관점)

- 괜히 버그바운티 프로그램을 오픈해서 더 위험한거 아닌가요?
    - 아니요. 서비스는 어차피 열려있고, 이미 누구에게나 공격 당할 위험이 있고, 제보된 리포트들로 직원들에게 최신 보안 공격 동향 교육으로 방어 전략을 세울 수 있어 장기적으로 보안성을 향상시킬 수 있다.
- 참여하는 윤리적 해컬르 믿을 수 있나요? 취약점을 블랙마켓에 팔 수도 있지 않나요?
    - 버그바운티 프로그램에는 여러 프로그램이 있는데  Private의 경우 윤리적 해커들에게 NDA(비밀유지서약)을 받음과 동시에 가상자산 거래소 수준의 인증을 진행한다. 또한 버그바운티 특성상 여러명이 참여하여 취약점 발견자가 보고하지 않아도, 다른 인원이 발견하여 보고할 가능성이 높다.

### 버그바운티 프로세스

프로그램 선택 → 스코프 확인 → 취약점 탐색 → 취약점 발굴 → 리포트 제출 → 리포트 리뷰 → 포상금 수령

### 버그헌팅을 하기 위해 필요한 것

1. 스스로 탐구하고자 하는 마음가짐
2. 개런티가 없는 투자에 대한 용기
3. 계속 의심하는 마음가짐
4. 한 단계씩 끝까지 진행할 각오

## 버그헌팅 프로그램 발전, 버그헌팅 프로그램 종류

### 버그헌팅의 시작 (해외)

NETSCAPE에서 최초로 버그바운티 프로그램이 시작되어 상금을 주겠다고했다. 그 활동은 제한적이고, 영향도 미미하여 큰 주목을 못 받았다.

그 후 구글, 페이스북이 줄줄이 버그바운티 프로그램을 시작하여 2012년부터 버그바운티 프로그램의 플랫폼(hackerone, bugcrowd 등등)이 등장하고 확산되었다.

### 버그헌팅의 시작 (국내)

2006년 KISA가 최초로 버그헌팅의 프로그램을 시작하여 2012년 KISA가 버그바운티 신고포상제를 시작하며 2021년에 다양한 국내 버그바운티 플랫폼을 오픈하고, KISA 보안 취약점 정보 포털을 신규 오픈했다.

하지만 해외에 비해 국내의 버그바운티 발전은 느린편이다.

그래도 네이버, 카카오 등이 버그바운티를 운영중이며 플랫폼들도 확산되고 있는 추세이다.

### 버그헌팅 공격/탐색 수단에 대한 발전

수동적 접근법 → 자동화 기술의 발전 → 커뮤니티를 통한 확산 → AI 기반도구의 발전

### 버그헌팅 프로그램 종류

- VDP → 포상금이 없고, 취약점 공개 정책을 준수해야함
- BBP → 포상금이 있고, public 또는 private 선택이 가능하다.

#### 버그바운티 프로그램 종류

Public, Private, Invitation-only로 나눌 수 있다.

왼쪽으로 갈 수록 취약점이 많이 발견된 확률(양)이 높지만 오른쪽으로 갈 수록 신뢰된 해커들에게 취약점을 제보받을 확률이 높다.(질)

- Public
    - 참여대상: 보안 연구자, 해커, 학생, 일반인 등 누구나 참여
    - 범위: 해당 기업이 운영하는 모든 제품 및 서비스
    - 장점: 누락된 보안 취약점 발견, 기업의 보안 의식 강화와 이미지 향상
- Private, Invitation Only
    - 참여대상: 신뢰할 수 있는 인증된 전문자들만 초대하여 참여
    - 범위: 일반적으로 특정 제품이나 서비스를 대상
    ex) Github Private Repository
    - 장점: 신뢰도가 높은 전문가들로부터 직접 피드백 받음, 특정 제품 및 서비스에 집중적인 보안 강화
    - Private과 달리 Invitation Only는 외부에 버그바운티 진행 사실을 알리면 안 된다.

## 취약점 공개정책, 버그헌팅 플랫폼 종류

### 취약점 공개 정책 (VDP)

→ 윤리적 해커에게 보고를 위한 명확한 지침을 제공하기 위한 것

- 사이버 환경에서의 취약점 보고의 문제점
    - 적절한 담당자에게 연락하기 어려움
    - 적절한 연락 채널을 찾기 어려움
    - 보고 내용이 적절한 담당자에게 전달되었는지 확인이 어려움
- 취약점 공개 정책 부재로 인한 결과
    - 취약점 정보 유출
    - 취약점이 보고되지 않음
    - 기관이 취약점에 대한 정보를 얻지 못함
- 미 국방부에서는 미국 정부를 위해 만든 VDP를 개발함
- CISA (Cybersecurity and Infrastructure Security Agency)는 모든 기관들에게 취약점 공개 정책(VDP)를 설립하도록 지시했음
- VDP 템플릿
    - 연락처 정보: 취약점 보고를 어디로 하면 좋을지에 대한 정보
    - 보고 규칙: 어떤 유형의 취약점을 어떻게 보고해야 하는지 등에 대한 정보
    - 법적 보호: 취약점을 보고하는 연구자가 법적인 행동에 대해 보호받을 수 있도록 하는 규정
    - 보상 정책: 필요한 경우, 보고된 취약점에 대한 보상이나 인정에 대한 정책
    - 피드백: 정기적인 피드백 및 투명한 프로세스 공개
- VDP 5가지 구성요소
    - 목적/약속선언 - VDP가 왜 필요하고 중요한 이유에 대한 내용을 포함해야 함
    - 범위 (Scope) - 제품 및 서비스, 취약점 유형, 도메인 범위 등의 내용을 포함해야 함. 취약점을 찾는 사람들이 어떤 자산에 주의를 기울여야 할지에 대한 가이드라인 제공
    - 법적 보호 (Safe Harbor) - 찾아낸 취약점에 대한 법적 조치나 처벌을 받지 않을 것이라는 확신을 주는 선언
    - 프로세스 설명 - 취약점 찾아낸 사람들이 어떻게 보고서를 제출하고 제출 시 필요한 정보는 무엇인지에 대한 설명
    - 보고서 평가 방법 - 보고서가 어떻게 평가될 것인지에 대한 설명
    ex) 제출 후 첫 응답까지 대기 시간, 취약점을 언제부터 공개할 수 있는지
