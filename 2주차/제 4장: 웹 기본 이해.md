#### 웹 → HTTP 프로토콜을 이용하여 정보를 공유하는 서비스

#### 웹 서버 → 정보 제공자

#### 웹 클라이언트 → 정보를 받는자

#### HTTP → 웹상에서 서로 통신을 하기 위해 정해둔 프로토콜

### WEB

- FRONT-END - 사용자에게 보여지는 부분으로써 웹 리소스, 즉 웹의 자원으로 구성됨
- BACK-END
- WEB RESOURCES
    - HTML
    - CSS
    - JS
- URI
    - scheme - 자원의 어떤 프로토콜을 이용해 접근해야 하는지를 나타냄 (ex. http, https 등)
    - userifo - 일부 서비스에서 필요로 하는 사용자의 이름과 비밀번호 (거의 생략)
    - host - 접근하려는 자원이 위치한 서비스의 도메인 이름이나 IP 주소
    - port - 서버에서 접근하려는 자원을 제공하는 서비스의 포트 번호
    - path - 서버상의 자원의 경로
    - query - 서버에 전달하려는 추가적인 정보
    - fragment - 웹페이지의 문서 내에서 특정 부분을 가리키는 식별자 (보통 #을 붙임)
- HTTP
    
    → 서버와 클라이어트의 데이터 기원을 요청과 응답의 형식으로 정의한 프로토콜
    
    - HTTP의 헤드의 각 절은 CRLF로 구분된다. Start Line과 Header로 나뉨
    - HTTP의 body는 헤드의 끝을 나타내는 CRLF 뒤에 오는 모든 줄을 말함.
    클라이언트가 서버에 전달하려는 내용을 여기에 담음.
- DNS (Domain Name System)
    
    → 인터넷에서 도메인 이름을 IP 주소로 변화한 서비스이다.
    사람이 읽을 수 있는 도메인 이름을 컴퓨터와 ESC는 IP 주소로 변화하여 인터넷 자원에 대한 액세스를 용이하게 하는 데의 역할을 한다.
    
    - A Record
        - 도메인에 각각 개별의 IP가 매핑된 것
        - www.google.com → 172.217.24.68
        - mail.google.com → 142.250.199.69
    - CNAME
        - 하나의 도메인에 다른 이름을 부여하는 방식으로 별칭이라는 표현으로 많이 사용됨
        - blog.domdomi.world → domdom.tistory.com

## 클라이언트와 서버 통신원리 이해

### Web Basic Structure

간단한 프로세스: 사용자가 컴퓨터로 브라우저를 켠다 → 웹 브라우저에서 웹 페이지를 찾아 클릭하면 그 요청은 회사의 서버로 전달됨

### Request

GET 요청은 클라이언트가 서버로부터 정보를 요청하는 가장 일반적인 방법이다.

사용자가 웹브라우저의 주소창에 URL을 입력하고 전송하는 것으로부터 시작된다.

이것을 쿼리 파라미터라고 부르고, URL 끝부분에 물음표를 추가하고 키는 각 형태로 데이터를 표시한다.

- GET 요청 장점
    - 웹브라우저는 자동으로 GET 요청의 결과를 캐싱한다.
    - 브라우저의 히스토리에 기록된다.
    - 서버 입장에서 안전한 메서드라는 것 - 서버의 상태를 변경하지 않기 때문
    - 북마크가 가능하다.

### Response

응답은 주로 3가지 부분으로 규정된다.

- 상태코드
    - 응답의 첫 부분에 위치하며 서버와 클라이언트의 요청을 어떻게 처리했는지를 나타내는 3가지의 숫자 코드이다. (ex. 200, 404)
- 헤더
    - 응답에 대한 메타데이터를 제공하는데 이 정보를 통해서 클라이언트는 응답을 올바르게 해석하고 처리할 수 있다.
- 응답 본문
    - 클라이언트가 요청한 실제 데이터를 담는다.

### Methods

POST 메서드는 클라이언트에서 서버로 데이터를 전송하는데 주로 사용되며 주로 사버 상의 리소스를 생성하거나 서버의 상태를 업데이트하는 용도로 사용된다.

크기가 큰 데이터를 전송하거나 서버의 상태를 변경하려는 경우에 주로 사용되며 GET 메서드와는 다르게 역동성을 가지지 않아 같은 요청을 여러 번 보내면 서버의 상태가 여러 번 바뀔 수 있다.  

- GET - 데이터를 읽거나 검색할 때 사용
- POST - 데이터를 등록할 때 사용
- PUT - 데이터를 대체할 때 사용
- DELETE - 데이터를 삭제할 때 사용
- OPTIONS - 대상 서버에서 통신 기능 옵션을 반환
- TRACE - 대상 서버에 대한 경로를 따라 메시지 루프백 테스트를 수행

### Cookie & Session

일반적으로 사용자가 웹사이트에서 로그인 절차를 거치게 되면 아이디와 비밀번호를 입력해 서버에 로그인 요청을 보내게 된다. 

이 요청을 받은 서버는 사용자의 로그인 정보를 확인하고 검증에 성공했다면 setcookie라는 HTTP 헤더를 포함하는 응답을 보낸다.

이 **쿠키**에는 세션 아이디나 토큰과 같은 사용자를 식별할 수 있는 정보가 포함되어있다.

브라우저는 쿠키를 저장하고, 이후 사용자가 해당 사이트에 요청을 보낼 때마다 자동으로 포함되어 서버가 사용자를 의식할 수 있게한다.

여기서 HTTP Only라는 옵션은 쿠키가 자바스크립트를 통해 로드되는 것을 방지함으로써 스크립트를 통한 쿠키 조건을 제한하게 한다. (XSS 방지)

→ 버그 헌팅을 할 때 HTTP Only 옵션이 잘 설정되어 있는지, 우회할 수 있는 방법이 있는지 확인하자.

### User-Agent

→ HTTP 요청 중 하나로써 클라이언트의 종료, 버전 정보, 운영체제 등의 정보를 서버에 전달한다.

웹사이트가 User-Agent의 정보를 이용해서 모바일 브라우저로 접속한 경우와 데스크탑 브라우저로 접속한 경우를 구분하고 각각의 적합한 버전의 웹페이지를 제공한다. 

User-Agent는 브라우저의 기본 속성이지만 이를 변경하면 서버는 다른 환경의 클라이언트로부터 요청을 받은 것으로 인식하고 그에 따라 다른 능력을 제공할 수 있다. 
이걸 활용하면 버그 헌팅에서 장기적 버그나 보안적 허점을 찾는 데 활용할 수 있을 것이다.

## 클라이언트와 서버 통신 이해

### HTTP Status Code

- 200번대: 요청을 성공적으로 받은 상태
    - 200 OK: 요청이 성공되었고, 정보는 응답으로 반환
- 300번대: 요청 완료를 위해 추가 작업 조치가 필요한 상태
    - 302 Found: 요청한 리소스의 URL이 일시적으로 변경
- 400번대: 요청을 처리할 수 없는 상태
    - 400 Bad Request: 잘못된 문법으로 서버가 이해할 수 없음
    - 401 Unauthorized: 요청한 응답을 받기 위해 인증이 필요
    - 403 Forbidden: 리소스에 접근할 권리를 가지고 있지 않음
    - 404 Not Found: 요청받은 리소스를 찾을 수 없음
    - 405 Method Not Allowed: 요청한 메소드는 서버에서 제거되어 사용할 수 없음
- 500번대: 서버가 명백히 유효한 요청에 대한 충족을 실패한 상태
    - 500 Internal Server Error: 서버에 문제가 있지만 원인 파악 불가
    - 503 Service Unavailable: 서버가 요청을 처리할 수 없

### Proxy

→ 클라이언트와 서버의 트래픽을 중계

클라이언트는 원래 직접 서버로 보내려던 HTTP 요청을 Proxy로 보낸다. → 
Proxy는 이 요청을 받아 서버로 전달한다. (이 과정에서 Proxy는 요청을 수정하거나 로깅하거나 특정 규칙에 따라 요청을 차단하는 등의 작업을 수행할 수 있다.) → 
서버는 Proxy로부터 요청을 처리하고 HTTP 응답을 Proxy로 보낸다. →
Proxy는 서버의 응답을 받아 클라이언트로 전달한다. (이때도 수정, 로깅 가능)

Proxy를 사용하는 클라이언트와 서버 사이의 통신을 중간에서 제어하고 모니터링할 수 있으므로 일반적인 Proxy서버를 사용하는 형태였다면 보안강화, 성능향상, 로깅 및 디버깅 등 다양한 이점을 얻을 수 있을 것이다.

## 인증, 인가, 입력값 검증 등의 보안 매커니즘 이해

### 인증

- 인증방법
    - 사용자명과 비밀번호
    - 이메일
    - 휴대폰 번호
    - 생체인증
    - 이중인증
    - 공인인증서

데이터 전송 암호화를 안 했거나, 약한 비밀번호를 가졌거나, 안전하지 않은 세션관리를 가지고 있는 웹사이트라면 버그 헌터들에게 좋은 먹잇감이 될 수 있다.

### 세션 관리

- 쿠키
    - 사용자의 브라우저에 저장되는 작은 데이터 조각으로, 세션 ID와 같은 사용자를 식별하는 정보를 담고 있다.
    - 사용자가 웹사이트에 로그인하면, 그 사이트는 쿠키에 사용자의 로그인 정보를 저장하여 다음 방문 시 자동으로 로그인하도록 할 수 있다.
- 세션
    - 서버 측에서 사용자 정보를 저장하는 방법
    - 세션 ID가 사용자의 브라우저에 저장되고, 이 ID는 서버에서 사용자의 세션 데이터를 조회하는데 사용된다.
    - 쿠키보다 보안적으로 우수하다. 왜냐하면 사용자의 중요한 정보는 서버에 저장되고, 클라이언트에는 세션 ID만 저장되기 때문이다.
- 과정: 
사용자 로그인 → 
서버에서 사용자 DB를 통해 사용자 확인 → 
서버에서 회원 정보 세션을 생성하여 세션 Storage에 저장 → 
서버는 사용자에게 세션 ID를 발급 → 
세션 ID는 서버가 세션 Storage에서 사용자 정보를 찾아내기 위한 고유 식별자로 활용 → 
사용자는 서버의 특정 기능을 실행하기 위해 요청을 보낼 수 있게 됨(with 쿠키) →
서버는 이 쿠키값을 확인해 세션 Storage에 있는 세션 ID와 비교 → 
검증이 완료되면 서버는 사용자 정보(세션)를 획득하여 요청 처리 →
사용자에게 데이트를 응답
- 만약 쿠키가 탈취되면?
    
    → 공격자는 탈취한 쿠키값으로 서버에 요청을 보낼 수 있게 된다. 
    서버는 쿠키값에 담긴 세션 아이디를 통해 사용자를 인식함으로써 쿠키를 보낸 주체가 누구인지 알 수 없다. 
    → 이것을 Session Hijacking이라고 한다.
    

### 인증 관련 취약점

- Session Hijacking
    - 사용자 세션을 탈취하는 공격 방법. 공격자는 사용자의 세션 ID를 획득해 서버에 해당 사용자로서 행동할 수 있음. 권한 상승이나 기타 취약점에 이용될 수 있음.
- Bruteforce
    - 보안 시스템을 무력화하기 위해 모든 가능성 있는 조합을 시도하는 공격 방법. 예) 패스워드 또는 해시 크래킹이 있으며 공격자가 시스템에 접근하기 위해 가능한 모든 조합을 시도해보는 공격 유형임.
- MFA Bypass
    - 휴대폰인증, 지문인증, OTP 인증 등의 다중 요소 인증을 우회하여 시스템에 접근하는 것을 말함.
- Password Reset
    - 사용자 비밀번호를 재설정하는 프로세스를 악용하여 타 사용자의 계정을 탈취하는 것을 말함 (XSS or CSRF)

### 인가

- 인가
    - 해당 사용자가 어떤 리소스가 서비스에 접근할 수 있는 권한을 가지고 있는지 확인하는 과정. 시스템 내에서 사용자가 수행할 수 있는 작업의 범위를 정의함.
- RBAC
    - 사용자에게 역할을 부여하고, 이 역할에 따라 리소스에 대한 접근 권한을 결정.

### 인가에 사용되는 기술들

- JWT
    - JSON Web Tokens는 JSON 기반의 표준
    - 이 토큰은 두 개체 사이에서 정보를 안전하게 전송하는데 사용될 수 있음.
    - JWT는 사용자의 인증 상태와 권한을 나타내는데 사용되는 정보를 포함할 수 있음.
    - 헤더, 페이로드, 서명(base64 encoding)으로 이루어짐
    - 인증 메커니즘에서 세션 정보를 저장하는 과정이 필요가 없다. (사용자의 정보와 이 정보의 무결성을 증명하는 서명 정보가 모두 토큰 안에 들어있기 때문(Stateless))
- OAuth
    - 사용자가 서비스를 제3의 애플리케이션에 위임할 수 있게 하는 표준
    - ex) 사용자가 Goolge 계정으로 다른 웹사이트에 로그인하면, 그 웹사이트는 Google의 OAuth를 사용하여 사용자를 인증하고 인가함.
    - OAuth가 사용되기 전에는 인증방식의 표준이 없어서 기존 기본인증인 아이디와 비밀번호를 사용하였는데, 이는 보안상 굉장히 취약하다.
    - → 하지만 OAuth를 통해 여러 애플리케이션을 통합하여 사용하는 것을 가능하게 했고, 사용자 입장에서 서비스의 편의성을 크게 향상시켰고, 동시에 정보 보안도 강화되게 해주었다.
    - Resource Owner(사용자), Client, Resource Server, Authorization Server로 구성됨.

### 입력 검증 (Input Validation)

- 주요 목적
    1. 보안 강화
    2. 데이터의 품질을 보장하는 것
    3. 시스템의 안정성 유지
- ex) Directory Traversal 취약점

### Whitelist vs Blacklist

- Whitelist 기반 필터 → 특정 조건을 만족하니 입력만을 허용하고 그 외의 모든 것을 거부
- Blacklist 기반 필터 → 특정 조건을 만족하는 입력을 거부

보통 Whitelist가 더 안전하다고 하는데 그 이유는 알려진 유효한 입력만을 활용하므로 예상치 못한 위협을 효과적으로 처단할 수 있기 때문이다.

### Filtering

필터링은 입력과 검증의 중요한 부분이며 이를 통해 활용되지 않은 입력을 제거하거나 변환할 수 있다. 

ex) 정규식 기반 필터링, 범위 기반 필터링, 내장 함수 기반 필터
